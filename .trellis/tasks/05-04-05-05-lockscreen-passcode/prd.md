# Lock screen 4-digit passcode

## Goal

Add an optional 4-digit numeric passcode protecting the device when it
wakes from sleep. Default off; opt-in via Settings. Touch + button input.

## What I already know

- `SleepActivity` already handles wake; today it routes directly to
  Dashboard.
- `KeyboardEntryActivity` exists and supports touch + button input — it's a
  reference for the on-screen numeric keypad pattern.
- `CrossPointSettings` / `SettingsList.h` are the home for the new toggle
  + optional code storage. Settings already supports `Enum` and `Value`
  field types. A new `Action` may be needed for "Set / Change / Disable
  passcode".
- 24 i18n yaml files exist under `lib/I18n/translations/`. New `STR_*`
  keys added there, regenerated by `scripts/gen_i18n.py`.
- Mofei device specifics: physical 800×480 landscape panel, logical
  480×800 portrait, 8 MB PSRAM, FT6336U touch.

## Assumptions (temporary)

- Default off, opt-in. No forced enrollment.
- 4 digits is enough; no longer codes for v1.
- Wrong-code rate-limit and lockout are required for v1.
- Forgot-passcode path is OUT OF SCOPE for v1 (factory reset is the
  recovery path).

## Open Questions

- (Preference) Lockout policy on N wrong attempts:
  1. After 3 wrong attempts, block input for 30 s, then reset counter.
  2. Exponential backoff (10s → 30s → 5min → 1h).
  3. Permanent lockout requires SD-card factory reset.
- (Preference) Hash function for the stored secret:
  1. SHA-256 truncated to 16 bytes (uses `mbedtls_sha256` already linked).
  2. Plain salted MD5 (smaller, mbedtls also has it).
  3. Plaintext compare (NOT recommended; rejected unless user insists).
- (Preference) When is the lock screen shown?
  1. Every wake from sleep, regardless of duration.
  2. Only after sleep ≥ 5 min (short sleep is "screen off but not locked").
  3. Configurable timeout in Settings.
- (Blocking) Is there an existing passcode use elsewhere (e.g., admin
  unlock, web UI auth) that we should reuse the hash format from?

## Requirements (evolving)

- New `LockScreenActivity` shown on wake when `SETTINGS.lockScreenEnabled`
  is true.
- Numeric keypad UI: 0-9 grid (3×4 with `<-` and `OK`), 4 progress dots,
  status text area for hints / errors.
- Touch + button dual input: tap a key OR use D-pad to navigate +
  Confirm.
- Hash + salt stored in SETTINGS. Salt is per-device random, generated on
  first passcode set.
- Wrong-attempt counter persisted; lockout state survives reboot.
- Settings entries: enable/disable, change code, lockout policy
  (if exposed).
- 8–12 new `STR_*` keys; English + simplified_chinese + traditional_chinese
  authored, other 21 languages can fall back to English at first.

## Acceptance Criteria (evolving)

- [ ] Wake-from-sleep with passcode enabled shows `LockScreenActivity`
- [ ] Correct 4 digits → enter Dashboard
- [ ] 3 wrong attempts → input blocked per chosen lockout policy
- [ ] Setting "Disable passcode" requires entering current code
- [ ] Persist across reboot: passcode hash, salt, attempt counter
- [ ] Touch hits a key only when within its rect (no inter-key dead zone
      pollution)
- [ ] Button D-pad navigates the keypad and confirm-key activates the
      focused key
- [ ] No regression in existing wake-from-sleep flow when passcode is
      disabled (default)

## Definition of Done

- All quality gates: `pio run -e mofei`, `pio check`, `git diff --check`,
  device flash, manual lock/unlock cycle on hardware
- Spec compliance: passcode hash uses an mbedtls primitive, never stored
  in plaintext
- i18n: new keys present in all 24 yamls (machine-translated for non-CJK
  languages is acceptable as a stub)

## Out of Scope (explicit)

- Forgot-passcode recovery (no email, no security questions)
- Biometrics
- > 4 digit codes
- Per-activity locking (only sleep wake)
- Time-based auto-lock during use

## Technical Notes

- Reference activities for layout pattern: `KeyboardEntryActivity`,
  `ButtonRemapActivity`, `ConfirmationActivity`.
- Storage: SETTINGS already persists to SD `/.crosspoint/settings.json`.
  Adding three fields: `lockScreenEnabled (bool)`, `lockScreenSalt
  (16-byte hex)`, `lockScreenHash (32-byte hex)`.
- Wake hook: extend `SleepActivity::onResumeFromSleep()` (or whatever the
  exact name is — to confirm during implement) to start
  `LockScreenActivity` when enabled.
